## Decisions made

- All four planning decisions (rules hardening, delete-rule guard, owner cascade, patterns-only
  seed) were Zara-approved before implementation; logged in DECISIONS.md.
- Delete guard shipped as `projects_via_pattern.id = ""` rather than `:length` — `:length` is
  documented for stored array fields only and back-relations are join-resolved; the `= ""` form
  is definitely valid filter grammar, and rules-check arbitrates its semantics empirically.

## Verified (Zara, `logs/verify-fresh-1.1.txt`) — both acceptance boxes checked in PLAN.md

- `npm run verify:fresh`: fresh `pb_data`, both migrations applied cleanly, seed green,
  **rules-check 99/99 — twice** (sweep idempotency proven).
- Both empirical rule idioms resolved on v0.39.6:
  1. `projects_via_pattern.id = ""` — blocks delete while linked, permits it at zero links
     (empty back-relation match-all is vacuously true). One in-session fix: the guard must be
     set AFTER `projects` exists — a rule can't validate a back-relation to an uncreated
     collection, so it lands in a second `app.save(patterns)` (same trick as `resets_with`).
  2. `@request.body.pattern.owner/.visibility` traversal in projects.updateRule works: re-point
     to shared/own allowed, to a private pattern denied, link-later allowed.
